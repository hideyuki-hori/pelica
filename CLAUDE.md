# pelica

家計簿と支出シミュレーションを統合した個人向けデスクトップアプリケーション。

---

## 技術スタック

- **デスクトップ**: Tauri
- **フロントエンド**: Solid + TypeScript
- **バックエンド**: Tauri Rust 側で SQLite を扱う（DB 配置: `~/.pelica/db`）
- フロントからは Tauri のコマンド経由で DB アクセス

---

## 設計思想：時系列ストリーム + フィルタ

取引データを区切りのない連続したストリームとして保持する。

- 月・年・週などの区切りはデータ側に持たない
- 区切りはフィルタの段階で初めて発生する
- 「過去」と「未来」もストリーム上の連続体として扱う（実績と予定がシームレスに繋がる）

### range フィルタの動作

例: `range since:2026/1/1 until:2026/1/31`

1. **期首残高**: since の前日までの全イベントを畳み込んだ残高
2. **期間内取引**: since 〜 until に該当する取引一覧
3. **支払予定**: 期間内に発生する将来の支払（カード引き落とし等）
4. **予測支出**: until が未来日の場合、過去平均ベースの予測支出を自動投影
5. **期末残高**: 期首残高 + 期間内収支 + 支払予定 + 予測支出

### What-if スイッチ
- 各予定項目・予測項目は個別に on/off 可能
- 家計簿機能とシミュレーション機能を一体化させる中核仕様

### 相対指定
- `range last:30d` のような相対指定もサポート予定（構文は実装段階で決定）

## データモデル：イベントソーシング + CQRS

### 採用理由
- 時系列ストリーム思想（区切りはフィルタで発生）と整合する
- 取引種別 × ステータス × 親子関係 × 口座種別 × リボ/分割の組み合わせ爆発を避け、責務を分離できる
- 後から射影を導入するのは困難なため、最初から射影前提で書く

### イベントの基本原則
- **追記のみ**: イベントログには追加のみ。過去のイベントは書き換えない
- **物理削除 NG**: 訂正・取消もすべてイベントとして記録する
- **イベントにバージョン番号**: 将来のスキーマ進化に備える

### イベントの粒度：業務粒度（ドメインイベント）

操作粒度（TransactionAdded 等）ではなく業務粒度を採用。
イベントログを見たときに「何が起きたか」を読み取れることを優先する。

イベント一覧（フィールド定義は [`docs/events.md`](./docs/events.md) を参照）:
- `IncomeRecorded` 収入があった
- `ExpenseRecorded` 支出があった
- `TransferExecuted` 振替した
- `CardPurchased` カードで買った
- `CardPaymentScheduled` 締め日が来て支払予定が確定した
- `CardPaymentExecuted` カード引き落としが行われた
- `LoanContractStarted` ローン契約を開始した
- `LoanRepaymentExecuted` ローン引落しが行われた
- `LoanPrepaid` 繰上返済を行った
- `LoanInterestRateChanged` ローン金利が変更された
- `TransactionCorrected` 記録を訂正した（旧値 → 新値）
- `TransactionVoided` 取り消した

### 射影（Projection）

書き込み（イベントストア）と読み取り（射影テーブル）を分離する。

**MVP に含める射影**:
1. 残高射影（口座ごとの現在残高）
2. カード支払予定射影（締め日・支払日を考慮した将来の支払予定）
3. 分割払い残回数射影
4. ローン残元本射影（残元本・残利息・残回数）

**MVP に含めない射影**:
- 取引一覧（イベントから直接生成しても十分速い見込み）
- カテゴリ別集計（range が動的なので動的計算で対応）

射影は対応するイベントから自動更新される。スキーマ変更時はイベント全件再生で再構築する。

## エンティティ概要

### Account（口座）
- 現金・銀行・カードなどすべての残高保持主体を統一的に表現
- 種別フィールドを持つ（cash / bank / card 等）

### Card（カード固有情報）
- Account とは別テーブルとして紐づける
- 締め日、支払日、支払元口座（紐づいた銀行口座）
- リボ・分割をサポートする方針

### Loan（ローン契約）
- 借入元金、借入期間、開始日、引落口座
- 金利は変更履歴を持てる構造（MVP では固定金利のみ運用、変動金利を後付けで追加できる土台を最初から確保）
- 返済方法は元利均等のみ MVP 対応（元金均等は将来）
- ローン種別フィールド（housing / auto / education / consumer 等）

### Category（カテゴリ）
- ユーザーがカスタマイズ可能なエンティティ
- 固定列挙型ではない

### Transaction 系
- 取引そのものはイベントから射影で復元される
- 種別: 収入 / 支出 / 振替 の 3 独立種別

### ScheduledTransaction（予定取引）
- 実績 Transaction とは概念として別管理
- 親 Transaction（カード購入など）から子の ScheduledTransaction 群が生成される
- 確定したらイベント（CardPaymentExecuted 等）が発火する

## 機能仕様（MVP）

### 取引種別
- **収入** / **支出** / **振替** の 3 独立種別
- 振替は from 口座 / to 口座 / 金額 の 1 取引として表現（ペア表現ではない）
- 理由: 意味論の一致、整合性、集計の素直さ、シミュレーションでの扱いやすさ

### 口座とフィルタ
- すべての取引はいずれかの口座に紐づく
- フィルタで `account:銀行A` のように指定するとその口座の取引だけ集計
- 指定なしなら全口座を合算

#### 振替のフィルタ動作
- `account:銀行A` で振替を見るとき、銀行 A から出た側か入った側かの「方向」を表示
- 同じ振替取引でも視点となる口座によって流出/流入が変わる
- フィルタなしの場合、振替は表示するが収入・支出の集計には含めない

### カード支払計算（連鎖モデル）
- カードを「締め日に集約され、支払日に銀行口座から引き落とされる」連鎖でモデル化
- カードでの支払 = カード口座の残高がマイナス方向に動く
- 締め日になると、その時点までの利用額が支払予定として確定
- 支払日になると、紐づいた銀行口座からカード口座への振替が発生

#### 仕様詳細
- **リボ・分割をサポート**: 詳細仕様は実装段階で詰める
- **休日繰越を考慮**: 締め日・支払日が休日の場合は翌営業日にずらす
- **ポイント還元は MVP 外**: ただし将来追加可能な設計にしておく

### ローン
- **対応するローン種別**: 住宅・自動車・教育・消費者金融などの月次返済型ローン
- **返済方法**: 元利均等のみ MVP 対応（元金均等は将来）
- **金利**: MVP では固定金利のみ運用する。ただしデータモデルは変動金利を後付けで追加可能な構造とし、金利変更は `LoanInterestRateChanged` イベントで記録できる前提を最初から含める
- **繰上返済**: 一部繰上返済をサポート（残元本を減らし、残期間または残月額を再計算。全額繰上返済はその特殊ケース）
- **休日繰越を考慮**: 引落日が休日の場合は翌営業日にずらす（カードと同じ）
- **MVP 外**: ボーナス払い、団体信用生命保険、抵当権処理

### 通貨
- MVP は円（JPY）のみ
- データモデル上、各取引・各口座に ISO 4217 通貨コードを持たせる
- MVP では全て JPY 固定、UI に通貨選択は出さない
- 将来の拡張時に通貨バリエーション・為替レート機能を追加

### 画面構成
- メイン画面は **ダッシュボード**
- 詳細な画面構成・遷移は画面設計フェーズで決定

## データ永続化方針

- 真実のソースは各端末の SQLite
- 端末間同期は **MVP では実装しない**
- 将来ホームサーバーへ移行する可能性あり。その時点で API 化を検討（MVP では抽象化しない）
- 個人の取引データ・残高は絶対にリポジトリに含めない（ローカル DB に閉じる）

## 開発フロー

- **issue ごとにブランチを切る**: `main` への直接コミットはしない
- **ブランチ命名**: `issues/<number>` 形式（例: `issues/5`）
- **コミットメッセージ**: `<issue-title> (#<issue-number>)` 形式（例: `scaffold tauri + solid (#2)`）
- **PR 本文**: `Closes #N` を入れて issue を紐付ける
- **マージ方式**: `gh pr merge --rebase --delete-branch`（線形履歴 + ブランチ整理）
- **雛形生成のような大量ファイル変更は 1 コミットで残す**: 後から「自分が何を変えたか」が diff で見やすくなる
- コミット粒度は論理的に意味のある単位（機械的な「1 ファイル 1 コミット」はしない）
